Customising Habitat

Customising habitat can be done using the include Habitat.yaml file.

The options are.

do_default_site:

This can be true or false. In the default yaml file it is set to true, and controls whether the zen.local domain is configured with a checkout of the v160 github code.

Unless you want to customise the way the code chackout is done, there is little reason to change this setting

do_default_map:

This can be true or false. In the default yaml file it is set to true, and controls whether to share the internal habitat directory with the host machine.

Again there is little reason to change this setting unless you have advanced knowledge of managing a vagrant virtual machine.

sites:

This option can be used create other virtual domains that map to a directory and optionally clone a github repository into that directory

Here's an example

                sites:
                  - skeleton: apache.default
                    domain: v150.local
                    dir: v150
                    github: https://github.com/zencart/zc-v1-series.git
                    branch: v150-release

The options within the sites directive are :-

skeleton:

In order to provision apache correctly, we use a skeleton vhosts file.

These skeleton files are stored in the shared scripts/skeletons directory.

There is currently only one skeleton file (apache.default). However others may be made available, or you may want to create your own.

domain:

This is the name of the domain that you want to map to your shared site. Currently, you will also have to create an entry in your hosts files for the domain.

dir:

This is the directory, relative to the shared habitat/web directory, that you want to be the web root of your site.

github:

This directive is optional, but if set will be used aas a directory to execute a git clone with.

branch:

This directive is optional, but if set will be used to execute a git checkout command with.
